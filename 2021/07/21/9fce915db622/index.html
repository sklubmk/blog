[알고리즘] 탐욕법(Greedy)과 구현(Implementation)

1. 당장 좋은 것만 선택하는 그리디

그리디 알고리즘은 어떤 문제가 있을 때 단순하게 탐욕적으로 문제를 푸는 알고리즘이다.

나중을 고려하지 않고 현재 상황에서 지금 당장 좋은 것만 고르는 방법으로, 마시멜로 실험 에 비유할 수 있다.

물론 이러한 점 때문에 별도의 조건이 없다면 최적해를 보장하지 않는다.



1-1. 동작 방식

그리디에는 딱히 정형화된 동작 방식이 없다.

앞서 말한대로, 문제마다 매 순간 최적의 해만 찾아가는 방식이다.

예를들어 A,B,C 노드가 일렬로 이어진 길에서 AC 까지의 최단 거리를 구하려면 AB 까지의 최단거리와 B~C까지의 최단거리의 합을 구하면 된다. 그러나 이 때에도 A에서 C를 잇는 다른 최단거리가 존재해서는 안된다는 등의 제한사항이 존재해야 한다.

즉, 문제마다 적용방법이 다르고 동작 방식도 다르다.



1-2. 어떤 경우에 동작하는가

그리디 알고리즘은

  • 탐욕 선택 속성(greedy choice property)
  • 최적 부분 구조(optimal substructure)

특성을 가지는 문제들을 해결하는데 강점이 있다. 매 선택이 다음 선택에 무관한 독립적인 값이어야 하며 매 순간의 최적해가 문제에 대한 최적해여야 한다는 의미이다.


[최적 부분 구조]
문제의 최적 해결 방법은 부분 문제에 대한 최적 해결 방법으로 구성되어야 한다는 뜻이다.


보통 창의력을 요하는 문제에서나, 순서에 대한 요구가 있을 때(정렬), 탐욕적으로 접근해도 정확한 답을 찾을 수 있다는 보장이 있을 때, 문제 풀이를 위한 최소한의 아이디어와 이가 정당한지 검토할 수 있을 때 사용 가능하다.



1-3. 근사 알고리즘

그리디는 최적해를 보장하지 않고 어느정도 적합한 수준의 해답을 알려준다. 따라서, “적당히 괜찮은 방법”을 찾을때에 사용할 수 있다. 그러나 “적당히” 괜찮기 위해선 정확한 증명이 수반되어야 한다. 또한, 해답을 찾아가는 과정에서 그리디로 구한 값을 비교값으로 설정할 수 있다.



1-4. 사용 예시

  • AI에 있어서 결정 트리 학습법(Decision Tree Learning)
  • 활동 선택 문제(Activity selection problem)
  • 거스름돈 문제
  • 최소 신장 트리(Minimum spanning tree), 크러스컬 알고리즘
  • 제약조건이 많은 대부분의 문제
  • 다익스트라 최단거리 알고리즘
  • 허프만 코드(데이터 압축)




2. 아이디어를 코드로 바꾸는 구현

구현이란 ‘머릿속의 생각을 코드로 예외나 실수 없이 표현하는 것’이다.

여기엔 ‘피지컬’이라 불리우는 언어 문법 능숙도나 코드 작성 속도 등의 기본기가 어느정도 필요하다.

따라서 구현 문제는 피지컬을 요구하는 문제라고도 할 수 있다.



2-1. 필요 능력

앞서 설명한 ‘피지컬’을 키우기 위해선, 아래와 같은 능력들이 요구된다.

  • 언어 패러다임 이해도
  • 언어 문법 이해도
  • 문제 파악 능력
  • 라이브러리 숙련도
  • IDE 숙련도
  • 타자속도

이는 많은 알고리즘 문제를 접하면서 자연스럽게 증가하는 부분이지만, 단기간 빠르게 상승시키기 위해서는 지금 내게 부족한 능력이 무엇인지 파악하고 집중적으로 암기나 연습을 할 필요가 있다.



2-2. 구현 시 고려해야 할 점

메모리 관리는 언어마다 차이를 보이는데, 직접 메모리를 관리해야하는 C나 C++에서는 훨씬 난이도가 높아진다. 그러나 이럴 필요가 없는 언어(대표적으로 파이썬)에서는 이 점에 대해선 크게 신경 쓸 필요는 없다.

단순 ‘코딩테스트’만을 위한 것이라면 사용 언어를 바꾸는 것이 꽤나 핵심적인 도약 방법이 된다. 특히 파이썬은 문법 작성이 매우 간결화 될 수 있으므로, ‘구현’에 최적화 되었다고 볼 수 있다.

문제의 제한 사항을 유심히 살펴 채점 환경을 고려하는 것도 중요하다. 파이썬 기준 기본적으로 10,000,000개 이상의 데이터를 가진 리스트는 메모리 오류가 날 가능성이 높다. 물론 이쯤되는 데이터를 사용하면 메모리 오류보다 시간제한에서 먼저 문제가 생긴다. 만약 사용 데이터가 크지 않아 제한 사항을 신경쓰지 않아도 된다면 구현 문제일 가능성이 높다.

문제의 길이가 매우 길고 복잡하면 지레 겁먹을 수 있는데, 이는 오히려 쉽게 구현으로 풀이되는 문제일 경우가 많다.



2-3. 인덱스 관리 능력

인덱스 관리 능력은 단순하지만 나름 피지컬 능력에 속한다.

보통 문제에서 주어지는 노드번호는 1로 시작하는 경우가 많다. 이를 배열 인덱스로 사용하려면 각 번호에 -1을 반드시 해줘야 하는데 여간 번거로운 작업이 아닐 수 없다. 코드가 길어지며 인덱스 사용 빈도가 높아진다면 더욱 실수할 가능성이 높아진다.

이를 쉽게 관리하는 방법에는 여러가지가 있고, 그 중 일부는 아래와 같다.

  1. arguments 로 제공된 데이터의 인덱싱

    • 문제 상단에서 -1을 선언하고 시작한다

  2. 노드 번호로 제공된 데이터의 인덱싱

    • 이를 저장할 배열 크기에 +1을 한다

이외에도 인덱스 관리 능력이 필요한 여러 경우들이 있다.

  1. 행과 열이 정해진 2차원 배열의 관리
    • 관리 및 수정이 힘들 경우 1차원 배열에 지정하여 사용
  2. index가 포함된 반복문 문법 활용
    • python
    • javascript
